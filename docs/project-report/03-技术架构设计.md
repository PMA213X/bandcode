## 3.2 技术选型

| 类别 | 技术 | 选型理由 |
|------|------|----------|
| 大模型 | DeepSeek Chat (通过 OpenAI 兼容 API) | 性价比高，中文支持好 |
| LLM 客户端 | AsyncOpenAI (openai 库) | 异步调用，支持流式输出 |
| Embedding 模型 | SentenceTransformers (all-MiniLM-L6-v2) | 轻量级，384 维向量，多语言支持 |
| 向量数据库 | ChromaDB (PersistentClient) | 内置持久化，支持元数据过滤 |
| 后端框架 | FastAPI | 异步优先，自动 API 文档 |
| 流式通信 | sse-starlette | SSE 事件推送，支持 12 种事件类型 |
| 前端框架 | React + TypeScript | 类型安全，组件化开发 |
| 样式方案 | Tailwind CSS | 原子化 CSS，开发效率高 |
| 数据库 | SQLite | 轻量级，嵌入式，无需额外服务 |
| Memory 存储 | Markdown 文件 (.mimo/) | 六层分层存储，人类可读 |

## 3.3 FastAPI 应用架构

后端入口 `backend/main.py` 负责创建 FastAPI 应用实例并注册所有组件：

### 中间件链

```
请求 → SecurityHeadersMiddleware → RequestLoggingMiddleware → CORSMiddleware → 路由处理
```

- **SecurityHeadersMiddleware**: 添加安全响应头（X-Content-Type-Options、X-Frame-Options 等）
- **RequestLoggingMiddleware**: 记录请求方法、路径、处理时间等日志
- **CORSMiddleware**: 配置跨域资源共享策略（allow_origins=["*"]）

### 路由注册

所有 API 路由统一以 `/api` 为前缀注册：

| 路由模块 | 前缀 | 功能 |
|----------|------|------|
| chat_router | /api/chat | 流式聊天（SSE）、聊天历史 |
| settings_router | /api/settings | 获取/更新系统配置 |
| memory_router | /api/memory | 获取/搜索/压缩记忆数据 |
| project_router | /api/project | 初始化项目、获取项目状态 |
| tools_router | /api/tools | 获取工具列表、调用工具 |
| users_router | /api/users | 创建用户、获取用户列表 |
| test_router | /api/test | 模型连接测试、流式输出测试 |
| workspace_router | /api/workspace | 获取工作区信息、列出文件 |

## 3.4 工作流管线设计

项目采用自研的 Pipeline 工作流引擎（`backend/workflow/pipeline.py`），而非直接使用 LangGraph。Pipeline 定义了 8 个节点顺序执行：

```
节点1: Constraint → 节点2: RAG → 节点3: Prompt → 节点4: Planner →
节点5: Approval → 节点6: SubAgent → 节点7: Tester → 节点8: Review
```

### PipelineState 数据结构

`backend/workflow/state.py` 定义了贯穿整个工作流的状态数据结构：

```python
@dataclass
class PipelineState:
    # 输入
    user_input: str = ""
    session_id: str = ""
    project: str = ""

    # Constraint Agent 输出
    constraints: list[str] = field(default_factory=list)
    constraint_summary: str = ""

    # RAG 输出
    rag_context: str = ""

    # Memory 上下文 (六层)
    memory_context: dict = field(default_factory=dict)

    # Planner 输出
    plan: Optional[dict] = None
    # 结构: {"tasks": [...], "delegated_agent": "simple-coder", "reason": "..."}

    # 子 Agent 输出
    agent_output: Optional[dict] = None

    # Tester 输出
    test_result: Optional[dict] = None

    # Review Agent 输出
    review_result: Optional[dict] = None

    # 流程控制
    current_step: str = "init"
    approval_pending: bool = False
    retry_count: int = 0
    max_retries: int = 3
    error: Optional[str] = None
    done: bool = False
```

### Review 修正循环

Pipeline 支持 `run_with_review_loop` 方法，当 Review Agent 发现违规时，自动将违规信息反馈给 Planner 重新生成，最多重试 `max_retries`（默认 3）次：

```python
async def run_with_review_loop(self, state: PipelineState) -> PipelineState:
    for attempt in range(state.max_retries):
        state = await self.run()
        if state.review_result and not state.review_result.get("passed", True):
            state.retry_count += 1
            violations = state.review_result.get("violations", [])
            state.user_input = "请修正以下违规项：\n" + "\n".join(...)
            continue
        break
    return state
```

## 3.5 SSE 事件推送机制

`backend/api/sse.py` 实现了完整的 SSE 事件推送系统，定义了 12 种事件类型和对应的 Pydantic 数据模型：

### 事件类型定义

```python
class SSEEventType:
    AGENT_START = "agent_start"
    CONSTRAINT_RESULT = "constraint_result"
    PLAN = "plan"
    APPROVAL_REQUIRED = "approval_required"
    TOOL_CALL = "tool_call"
    CODE = "code"
    TEST_RESULT = "test_result"
    REVIEW_RESULT = "review_result"
    MEMORY_UPDATE = "memory_update"
    TEXT = "text"
    DONE = "done"
    ERROR = "chat_error"
```

### 事件推送流程

1. 聊天接口创建 `asyncio.Queue` 事件队列
2. 使用 `asyncio.create_task()` 启动后台处理任务
3. 处理函数通过 `push_*` 系列函数将事件放入队列
4. `sse_generator()` 异步生成器从队列读取事件，生成 SSE 格式字符串
5. `EventSourceResponse` 包装生成器返回给前端
6. `push_done()` 放入 `None` 作为结束信号
