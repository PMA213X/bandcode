## 3.3 Pipeline 工作流说明

BandCode 的核心是一个 **8 节点顺序执行的 Pipeline 工作流**，定义在 `backend/workflow/pipeline.py` 中。Pipeline 接收一个 `PipelineState` 数据结构，在各节点间传递状态，最终生成 AI 回复。

### PipelineState 数据结构

`PipelineState` 是一个包含 17 个字段的 dataclass，贯穿整个 Pipeline 执行过程：

| 字段 | 类型 | 说明 |
|------|------|------|
| user_input | str | 用户输入文本 |
| session_id | str | 会话 ID |
| constraints | list | Constraint Agent 检索到的约束列表 |
| rag_context | list | RAG 检索到的文档片段 |
| memory_context | dict | 各层 Memory 上下文 |
| plan | dict | Planner 生成的执行计划 |
| selected_agent | str | Planner 选择的 Agent 名称 |
| agent_output | str | 子 Agent 的输出结果 |
| test_result | dict | Tester 的测试结果 |
| review_result | dict | Review 的审查结果 |
| is_approved | bool | 审批是否通过 |
| error | str | 错误信息 |
| history | list | Agent 执行历史 |
| messages | list | 对话消息列表 |
| project | str | 项目名称 |
| tools_used | list | 已使用的工具列表 |
| status | str | 当前状态 |

### 8 个 Pipeline 节点

| 节点顺序 | 节点名称 | 执行内容 | 说明 |
|----------|----------|----------|------|
| 1 | 约束检索（node_constraint） | Constraint Agent 从 Memory 检索约束 | 为后续 Agent 提供编码规范、架构决策等参考 |
| 2 | RAG 检索（node_rag） | ChromaDB 语义检索相关文档 | 将用户输入向量化，检索 Top-K 相关知识片段 |
| 3 | Prompt 构建（node_prompt_build） | PromptBuilder 组装完整 Prompt | 整合系统提示词、记忆上下文、RAG 上下文、用户消息 |
| 4 | Planner 调度（node_planner） | Planner Agent 分析需求并选择 Agent | 分解任务，根据复杂度选择 SimpleCoder 或 ComplexCoder |
| 5 | 审批检查（node_approval） | 检查是否需要用户审批 | 高风险操作（删除文件等）需用户确认 |
| 6 | 子 Agent 执行（node_subagent） | Coder Agent 执行编码任务 | 使用工具系统完成代码生成/修改 |
| 7 | Tester 验证（node_tester） | Tester Agent 验证代码质量 | 编译检查、单元测试、静态分析 |
| 8 | Review 审查（node_review） | Review Agent 检查合规性 | 检查代码是否符合项目规范，不通过则触发 ReviewLoop |

### ReviewLoop 机制

当 Review 审查不通过时，`ReviewLoop`（`backend/workflow/review_loop.py`）会自动触发修复流程：

1. 将 Review 的反馈传递给 Coder Agent
2. Coder Agent 根据反馈修改代码
3. Tester 重新验证
4. Review 重新审查
5. 超过最大重试次数时，回滚到最近的快照

`CheckpointManager`（`backend/workflow/checkpoint.py`）在关键节点创建文件快照，支持回滚操作。

## 3.4 Agent 系统设计

### 六个 Agent 的职责

| Agent | 文件 | 角色 | 核心能力 | 权限 |
|-------|------|------|----------|------|
| Planner | `agents/planner.md` | 需求分析与任务调度 | 分析需求、分解任务、选择 Agent | read, edit |
| SimpleCoder | `agents/simple-coder.md` | 简单编码任务 | UI 调整、配置修改、单文件编辑、小 Bug 修复 | read, edit |
| ComplexCoder | `agents/complex-coder.md` | 复杂编码任务 | 核心业务逻辑、API 开发、架构变更、跨模块重构 | read, edit |
| Tester | `agents/tester.md` | 测试验证 | 编译检查、单元测试、静态分析 | read（只读） |
| Constraint | `agents/constraint.md` | 约束检索 | 从 Memory 层检索相关约束信息 | read（只读） |
| Review | `agents/review.md` | 审查修正 | 代码合规性检查、文件合规性检查 | read（只读） |

### Agent 自动升级机制

SimpleCoder 具备自动升级能力：当检测到任务复杂度超过阈值（如跨模块修改、架构变更）时，会自动将任务升级给 ComplexCoder 处理。这通过 `SimpleCoderAgent._should_upgrade()` 方法实现，避免了简单任务使用重型 Agent 造成的资源浪费。

### Agent 提示词管理

每个 Agent 的系统提示词存储在 `agents/*.md` 文件中，通过 Agent 子类的 `system_prompt` 属性加载。提示词定义了 Agent 的角色、职责、输出格式和行为约束，是 Agent 行为的核心驱动力。
