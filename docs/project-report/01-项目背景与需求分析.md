# 第一章 项目背景与需求分析

## 1.1 项目背景

熏掌门是武夷山非遗食品品牌，主要销售非遗熏鹅产品。目前主要通过微信、抖音、小红书等渠道进行销售。

在用户咨询过程中，大量问题高度重复，例如：
- 口味选择
- 储存方式
- 保质期
- 加热方式
- 物流时效
- 价格咨询

这些问题需要人工客服重复回答，效率低下，且无法保证回答的一致性。

## 1.2 用户痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| 重复问答 | 大量相似问题需要人工重复回答 | 客服效率低，人力成本高 |
| 响应慢 | 人工客服无法24小时在线 | 用户体验差，潜在客户流失 |
| 回答不一致 | 不同客服回答可能存在差异 | 品牌形象受损 |
| 缺乏数据 | 无法统计分析用户咨询内容 | 无法优化产品和服务 |

## 1.3 项目目标

1. 实现 7x24 小时智能客服，自动回答用户常见问题
2. 通过 RAG 技术降低大模型幻觉，确保回答准确性
3. 识别用户意图，自动调用相关工具完成业务操作
4. 记录对话日志，支持数据分析和优化
5. 提供友好的用户交互界面

## 1.4 服务人群

- 熏掌门的潜在客户
- 老客户复购咨询
- 通过微信、抖音、小红书等渠道咨询的用户
- 需要了解产品信息、下单、售后服务的用户

## 1.5 核心使用场景

| 场景 | 描述 | 示例 |
|------|------|------|
| 产品咨询 | 用户询问产品信息 | 「你们的熏鹅有哪些口味？」 |
| 价格查询 | 用户询问价格 | 「熏鹅多少钱一只？」 |
| 物流咨询 | 用户询问配送方式 | 「可以发顺丰吗？多久能到？」 |
| 储存方式 | 用户询问保存方法 | 「熏鹅怎么保存？能放多久？」 |
| 订单查询 | 用户查询订单状态 | 「我的订单到哪了？」 |
| 产品推荐 | 用户需要推荐 | 「送礼选哪个比较好？」 |
| 售后服务 | 用户反馈问题 | 「我收到的熏鹅有问题」 |

## 1.6 业务场景详细分析

### 产品咨询场景

熏掌门的产品线涵盖多种口味的非遗熏鹅产品，用户在选购时需要了解口味差异、规格参数、配料成分等详细信息。传统客服需要记忆大量产品知识，容易出现回答不一致的情况。智能客服通过 RAG 技术从知识库中精准检索产品文档，确保每次回答基于同一份权威资料。

### 物流配送场景

武夷山特产涉及冷链配送、保鲜包装等特殊物流需求。用户关心配送时效、运费标准、是否支持指定快递等问题。系统需要根据用户所在地区和购买产品类型，调用物流工具查询实时配送方案。

### 售后服务场景

食品安全类产品的售后处理尤为敏感，包括产品质量投诉、退换货申请、保质期争议等。系统需要识别用户的售后意图，自动调用订单查询工具获取订单详情，结合售后政策知识库给出规范化的处理建议，同时在超出处理能力时及时转接人工客服。

### 产品推荐场景

用户在送礼、团购、自用等不同场景下有差异化的产品需求。系统通过理解用户的使用场景、预算范围、口味偏好等信息，结合产品知识库进行智能推荐，提升转化率。

## 1.7 技术需求分析

基于上述业务场景，系统需要满足以下技术需求：

| 需求类别 | 具体需求 | 技术实现 |
|----------|----------|----------|
| 自然语言理解 | 准确识别用户意图（咨询/查询/推荐/售后） | LLM + 意图分类 Prompt |
| 知识检索 | 从产品文档中精准检索相关信息 | RAG + ChromaDB 向量检索 |
| 工具调用 | 查询订单、推荐产品等业务操作 | Agent + Tool 机制 |
| 流式响应 | 实时逐字输出 AI 回复，提升用户体验 | SSE (Server-Sent Events) |
| 多轮对话 | 保持上下文连贯，支持追问和澄清 | 会话历史 + Memory 系统 |
| 数据持久化 | 记录对话日志，支持回溯和分析 | SQLite 数据库 |
| 系统可观测性 | 实时展示 Agent 执行状态和工具调用过程 | 12 种 SSE 事件类型 |
